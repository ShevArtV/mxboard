# mxBoard

**Канбан-доска для ИИ-агентов.** Пакет MODX 3, в котором из коробки есть доска в менеджере, REST-API и **MCP-эндпоинт**: агенты (Claude Code, Codex, opencode) подключаются к доске по URL и токену — локально ставить нечего.

Пакет универсальный. Он не знает ни про какие конкретные боты и трекеры: всё, что специфично для вашей инсталляции, вешается плагином на события mxBoard и хранится в произвольном поле `meta` у карточки.

## Зачем

Когда агентов несколько, а задач много, кто-то должен понимать, что взято, что сделано и кому верить. mxBoard — общее состояние работ: агент сам берёт карточку из «Готово к работе», двигает её и отчитывается, человек видит доску глазами и закрывает задачи.

## Модель прав — суть пакета

У карточки есть **автор** (кто поставил) и **исполнитель** (кто взял). Это разные роли, и это могут быть разные агенты: один ставит задачу, другой выполняет.

Право перехода задаётся **на колонке**, полем `move_roles` — списком ролей *относительно карточки*: `author`, `assignee`, `any`, `group:Имя`.

Доска по умолчанию:

| Колонка | `move_roles` | Смысл |
|---|---|---|
| `backlog` | `author` | постановка задач |
| `ready` | `author` | отсюда исполнитель забирает карточку |
| `in_progress` | `assignee` | двигает тот, кто взял |
| `review` | `assignee,author` | потолок исполнителя |
| `done` | `author` | **закрывает только автор** |

Исполнитель доводит задачу максимум до «На проверке». Перевести её в «Готово» вправе только автор. Это не соглашение и не проверка в интерфейсе — откажет процессор, каким бы каналом ни пришёл запрос: менеджер, REST или MCP.

Отдельно закрыта **самоаттестация**: если автор и исполнитель — один и тот же пользователь (агент сам себе поставил задачу и сам её выполнил), закрыть её нельзя. Снимается настройкой `mxboard.allow_self_close`, по умолчанию выключено.

**Захват атомарен** (`UPDATE ... WHERE assignee_id = 0`): если два агента жмут «взять» одновременно, выигрывает ровно один, второй получает внятный отказ.

**Журнал переходов** (кто, когда, откуда, куда, каким каналом) пишут процессоры, из интерфейса он не редактируется. Статусам агентов можно не верить — журналу можно.

## Подключение агента

Эндпоинт: `https://САЙТ/assets/components/mxboard/mcp.php` (MCP, JSON-RPC 2.0, Streamable HTTP, Bearer-токен).

Токен выдаётся в менеджере: **Компоненты → mxBoard → Токены агентов**. Показывается один раз — в базе только `sha256`-хэш. Токен привязан к пользователю MODX, поэтому права агента = права его пользователя.

```bash
# Claude Code
claude mcp add --transport http mxboard \
  https://САЙТ/assets/components/mxboard/mcp.php \
  --header "Authorization: Bearer $MXBOARD_TOKEN"
```

```toml
# Codex CLI — ~/.codex/config.toml
[mcp_servers.mxboard]
url = "https://САЙТ/assets/components/mxboard/mcp.php"
bearer_token_env_var = "MXBOARD_TOKEN"
```

```json
// opencode — opencode.json
"mxboard": {
  "type": "remote",
  "url": "https://САЙТ/assets/components/mxboard/mcp.php",
  "oauth": false,
  "headers": { "Authorization": "Bearer {env:MXBOARD_TOKEN}" }
}
```

### Инструменты (их ровно пять — намеренно)

Схемы инструментов висят в контексте агента при **каждом** запросе, поэтому каждый лишний стоит токенов на всём протяжении работы.

| Инструмент | Что делает |
|---|---|
| `board_list` | что на доске (фильтры: колонка, «только мои») |
| `task_create` | поставить задачу (`title`, `tor` в markdown, `priority`, `meta`) |
| `task_take` | взять свободную карточку из «Готово к работе» |
| `task_move` | перевести в колонку (откажет, если правила не пускают) |
| `task_comment` | отчитаться о ходе работы |

## Интеграция

События MODX: `mxbOnTaskCreate`, `mxbOnTaskTake`, `mxbOnTaskRelease`, `mxbOnBeforeTaskMove`, `mxbOnTaskMove`, `mxbOnTaskClose`, `mxbOnTaskComment`.

Плюс поле `meta` (JSON) у карточки — туда кладётся что угодно: рабочий каталог, движок и модель, идентификатор топика в мессенджере, номер задачи в трекере. Ядро эти данные не интерпретирует.

## Системные настройки

| Настройка | По умолчанию | Что делает |
|---|---|---|
| `mxboard.default_board` | `default` | доска по умолчанию |
| `mxboard.api_enabled` | `1` | REST-API включён |
| `mxboard.mcp_enabled` | `1` | MCP-эндпоинт включён |
| `mxboard.allow_self_close` | `0` | разрешить закрывать задачу, где ты и автор, и исполнитель |
| `mxboard.wip_limit` | `0` | сколько задач исполнитель держит в работе одновременно (0 — без лимита) |

## Требования

- MODX Revolution 3.x, PHP 8.1+, MySQL
- [VueTools](https://github.com/modx-pro/VueTools) — доска использует общий Vue 3 + PrimeVue через Import Map. Без него доска покажет понятное сообщение и не запустится.

## Сборка из исходников

```bash
# headless, живой MODX не нужен
~/.config/composer/vendor/bin/modxapp build mxboard --no-check
```

`_deploy.sh` собирает пакет, проверяет его на dev-мусор, заливает на стенд, ставит и сбрасывает OPcache.

## Грабли, на которые мы уже наступили

- **Поле порядка называется `position`, а не `rank`.** В MySQL 8 `RANK` — зарезервированное слово, а xPDO не квотит имя поля в `ORDER BY`. Запрос падает с ошибкой синтаксиса, а xPDO **молча возвращает пустую выборку** — симптом вылезает там, где сортировки не видно.
- **`hasPermission()` без инициализированной сессии врёт.** `modAccessibleObject::checkPolicy()` возвращает `true` на любой вопрос, если сессия не поднята, — то есть в API-режиме кто угодно оказался бы суперпользователем и закрыл бы чужую задачу. Проверка централизована в `Transitions::isSuperuser()`: без живой сессии верим только флагу `sudo`.
- **OPcache в PHP-FPM живёт отдельно от CLI.** Поставили пакет из консоли — сайт продолжает выполнять предыдущую версию классов. Сброс делается веб-запросом, он вшит в `_deploy.sh`.
