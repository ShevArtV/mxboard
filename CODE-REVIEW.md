# Code Review: mxBoard

**Дата:** 2026-07-17
**Версия:** 2.0.0-alpha
**Ревьюер:** MiMoCode

---

## Общая оценка: 8/10

Кодовая база выглядит продуманной и зрелой для альфа-версии. Архитектура с единым сервисным слоем (TaskService / StructureService / BoardQuery), над которым сидят три фасада (менеджер, REST, MCP) — сильное решение, исключающее расхождение прав между каналами. Автор явно понимал ловушки MODX (checkPolicy без сессии, `RANK` в MySQL 8, OPcache) и обошёл их.

---

## 1. Архитектура

### Сильные стороны

- **Единый сервисный слой.** `TaskService`, `StructureService`, `BoardQuery` — единственные владельцы бизнес-логики. Процессоры менеджера, REST-роутер и MCP-сервер — тонкие фасады. Расхождению прав между каналами взяться неоткуда.
- **Канальная идентификация.** Каждый переход пишет `channel` (`mgr` / `api` / `mcp`) — журнал показывает, каким путём пришло действие.
- **Права как данные, а не код.** `move_roles` на колонке — CSV строк; `Transitions::can()` читает его и матчит роль относительно карточки. Добавление роли — INSERT, а не патч кода.
- **Атомарный захват.** `UPDATE ... WHERE assignee_id = 0` — два агента не возьмут одну карточку.
- **Самоаттестация.** Запрет на уровне `Transitions::can()`, а не UI — сломать через API нельзя.

### Замечания

#### A1. Дублирование `deadline()` в MCP\Server и REST\Router

**Файл:** `core/components/mxboard/src/Mcp/Server.php:778-788`
**Файл:** `core/components/mxboard/src/Rest/Router.php:197-207`

Метод `deadline()` — точная копия `normalizeDeadline()` из `TaskService`. MCP и REST каждый держит свою копию. При изменении логики (новый формат даты) придётся править три места.

**Рекомендация:** MCP и REST должны передавать значение как есть — `TaskService` уже нормализует. Удалить `deadline()` из обоих фасадов.

#### A2. Дублирование бутстрапа в mcp.php и rest.php

**Файл:** `assets/components/mxboard/mcp.php:33-72`
**Файл:** `assets/components/mxboard/rest.php:27-70`

Идентичный блок поиска `config.core.php`, автолоадера и инициализации MODX. При изменении (новый путь, composer-autoloader) — два файла.

**Рекомендация:** Вынести в `bootstrap.php` (он уже есть в `core/components/mxboard/`), а в entry-point'ах — один `require`.

#### A3. `nextPosition()` — не монотонно и не потокобезопасно

**Файл:** `core/components/mxboard/src/Service/TaskService.php:652-658`

```php
private function nextPosition(int $columnId): int
{
    $c = $this->modx->newQuery(MxBoardTask::class);
    $c->where(['column_id' => $columnId]);
    return (int) $this->modx->getCount(MxBoardTask::class, $c);
}
```

При одновременном создании двух карточек в одну колонку обе получат `position = N`. При удалении карточки из середины — дыра. Позиции не переупаковываются.

**Рекомендация:** `MAX(position) + 1` вместо `COUNT(*)`, либо `ORDER BY position, id` при чтении и `position = createdon` (unix-время — монотонно).

#### A4. N+1 в `BoardQuery::username()` и `taskDetail()`

**Файл:** `core/components/mxboard/src/Service/BoardQuery.php:451-460`

`username()` делает `getObject(modUser)` на каждый вызов. В `taskDetail()` вызывается 2 раза (author + assignee), плюс для каждого субтаска и комментария. Для карточки с 5 подзадачами и 10 комментариями — 17 отдельных запросов.

**Рекомендация:** Кешировать в массив `$userId => username` на уровне вызова, или JOIN'ить при выборке.

#### A5. `StructureService::createProject` — колонки сохраняются вне транзакции

**Файл:** `core/components/mxboard/src/Service/StructureService.php:230-247`

Проект сохраняется, потом в цикле сохраняются колонки. Если на 3-й колонке `save()` упадёт — проект без колонок, а `templateColumns()` при следующей попытке вернёт ошибку «уже существует».

**Рекомендация:** Обернуть в `$this->modx->beginTransaction()` / `commit()` / `rollback()`. То же для `createType()` (тип + поля).

## 2. Безопасность

### Сильные стороны

- **checkPolicy-ловушка обойдена.** Живая сессия mgr + `session.use_cookies=0` + уничтожение в shutdown — правильно.
- **SHA-256 хэширование токенов.** Утечка таблицы не даёт рабочих ключей.
- **`hash_equals` для сравнения хэшей.** Тайминговый канал закрыт.
- **`isSuperuser()` — сначала `sudo`-флаг, потом `hasPermission()`.** Без сессии не прогалит.
- **Токен не логируется.** В журнал пишется `user_id` и `channel`, не секрет.
- **Самоаттестация** — на уровне сервиса, не UI.

### Замечания

#### S1. Сырой токен хранится в `profile.extended` (осознанный trade-off)

**Файл:** `core/components/mxboard/src/Processors/Mgr/Token/IssueForUser.php:82-86`

```php
$extended['mxboard'] = [
    'token' => $raw,
    'token_id' => (int) $token->get('id'),
    'createdon' => $now,
];
$profile->set('extended', $extended);
```

Сырой токен сохраняется в БД в открытом виде — это **намеренное решение**: в отличие от именованных токенов раздела «Токены агентов» (где хранится только хэш и токен показывается один раз), профильный вариант нужен для повторного копирования через UI. Trade-off: удобство (перекопировать токен без перевыпуска) vs. безопасность (дамп БД даёт рабочие ключи).

**Флаг, не блокер.** При желании усилить — AES-шифрование с ключом из настройки MODX, или отказ от хранения сырого токена (тогда — только показ один раз, как в Token/Create).

#### S2. Нет rate-limiting на MCP/REST

При переборе токенов или brute-force Basic-auth — каждый запрос = SELECT из БД + сессия. Нет ни IP-based throttling, ни блокировки по числу неудачных попыток.

**Рекомендация:** Минимум — fail2ban-friendly логирование неудачных попыток (401) в отдельный файл. Идеально — блокировка на N минут после M неудачных попыток (но это уровень reverse proxy или MODX-плагина).

#### S3. XSS: комментарии и ToR — проверить Vue-рендер

В Vue-интерфейсе (`TaskPage.vue`, `TaskCard.vue`) если `tor` или `content` рендерятся через `v-html`, а не `{{ }}`, возможен XSS от злонамеренного агента. Проверить, что Vue-шаблоны используют интерполяцию.

## 3. Качество кода (PHP)

### Сильные стороны

- **`declare(strict_types=1)`** везде.
- **PHPDoc с `@return array{...}`** — точные типы возврата.
- **Лексиконные ключи** вместо хардкода строк.
- **`fireEvent()` с try/catch** — кривой плагин не роняет доску.
- **Composite-связи** (комментарии и журнал удаляются каскадом с карточкой, поля — с типом).

### Замечания

#### C1. `log()` не проверяет результат `save()`

**Файл:** `core/components/mxboard/src/Service/TaskService.php:720`

```php
$log->save();
```

Если запись лога не сохранится (БД недоступна, переполнение) — молча. Журнал — ключевой аудит; тихая потеря — плохо.

**Рекомендация:** Хотя бы `$this->modx->log(modX::LOG_LEVEL_ERROR, ...)` при неудаче.

#### C2. `delete()` — raw SQL через `prepare()`

**Файл:** `core/components/mxboard/src/Service/TaskService.php:474-476`

```php
$stmt = $this->modx->prepare("UPDATE {$childrenTable} SET parent_id = 0 WHERE parent_id = :id");
$stmt->execute([':id' => $taskId]);
```

Параметризовано (не SQL-injection), но обходит xPDO и не обновляет кеши объектов. Если после `delete()` код прочитает те же объекты из кеша — получит старый `parent_id`.

#### C3. `BoardQuery::departmentUsers()` — raw PDO вместо xPDO

**Файл:** `core/components/mxboard/src/Service/BoardQuery.php:154-170`

```php
$c->prepare();
if (!$c->stmt || !$c->stmt->execute()) { return []; }
```

Работает, но стилистически выбивается: остальные запросы через `getCollection`. Причину понимаю (select подмножество полей + JOIN), но `select()` + `getCollection()` дал бы то же самое.

#### C4. `transferFlag()` — не атомарно

**Файл:** `core/components/mxboard/src/Service/StructureService.php:774-785`

Сначала снимает флаг со всех носителей, потом ставит на целевую. Если между `save()` старого и `save()` нового что-то упадёт — ни одна колонка не будет initial/final, и создание карточек сломается.

**Рекомендация:** Транзакция, или UPDATE одним запросом.

## 4. Фронтенд (Vue 3 + PrimeVue)

### Сильные стороны

- **Всё через `useLexicon`** — ни одного хардкода строк в компонентах.
- **Оптимистичное перемещение** (drag-and-drop) с откатом при отказе сервера.
- **Клиентская предпроверка** (`can_move_any`, `author_id`) — запрос не летит заведомо зря.
- **`useApi` из VueTools** — не свой велосипед.
- **Ленивая инициализация API** (`_api = null` → `api()`) — безопасно к порядку загрузки.

### Замечания

#### F1. `errorMessage()` — fallback строка на русском не из лексикона

**Файл:** `assets/components/mxboard/js/mgr/api/connector.js:49`

```js
export function errorMessage(err, fallback = 'Не удалось выполнить операцию') {
```

Дефолтный fallback — хардкод на русском. При английской локали MODX — непереведённая строка.

**Рекомендация:** `fallback = t('mxboard_err_generic')` или передавать из вызывающего кода.

#### F2. Нет обработки 401 на фронте

При истечении сессии MODX connector вернёт HTML-страницу логина (не JSON). `useApi` бросит с неинформативной ошибкой.

**Рекомендация:** Interceptor, который ловит non-JSON-ответ или 401 и показывает «сессия истекла, обновите страницу».

#### F3. `normalizeBoard` / `normalizeTask` — null-safety

Если сервер вернёт `null` вместо массива колонок (баг, пустой проект) — `normalizeBoard(res).columns` упадёт.

## 5. MCP-протокол

### Сильные стороны

- **Ровно 5 базовых инструментов** (+ структурные для менеджера) — экономия токенов в контексте.
- **`tools/list` фильтруется по правам** — исполнитель не видит структурных тулов.
- **`instructions` в `initialize`** — краткая инструкция агенту, как пользоваться доской.
- **Сессия только ради политик**, уничтожается в `shutdown_function`.

### Замечания

#### M1. `tools/list` не кешируется

Каждый `tools/list` вызывает `Transitions::isAnyDepartmentManager()` → N запросов к БД. MCP-клиенты часто запрашивают tools при старте сессии.
