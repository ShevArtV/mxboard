# mxBoard v2 — план: иерархия и формализация задач

> Статус: согласовано, к реализации. Составлен 2026-07-15.
> Продолжает работающий 1.0.0-alpha (доска + REST + MCP). Данные на стенде выкидные —
> схема пересобирается целиком, миграций нет.

## Что добавляет v2

Разворот из «одна доска» в **отдел → проект → задача**, с типизацией задач:
у каждого типа задачи свой набор обязательных полей (багфикс: где сломалось, что,
как воспроизвести, как должно быть). Плюс подзадачи (привлечение соисполнителей),
видимость по вовлечённости, дедлайны с оспариванием, фильтры, и агент-менеджер,
который через MCP создаёт задачи, типы и проекты.

---

## Принятые решения (и почему)

1. **Отдел = группа пользователей MODX**, но не любая. `Customers`, `Subscribers` —
   тоже группы, но не отделы. Поэтому тонкий реестр `mxboard_department` помечает,
   какие группы являются отделами. Членство, роли, супер-пользователь — всё из MODX.
2. **Проект = доска.** Отдельной сущности «доска» нет; колонки принадлежат проекту.
3. **Тип задачи обязателен.** Задач без типа нет — простая задача с одним заголовком
   ничем не лучше типизированной, а формализация и есть цель.
4. **Тип обязан иметь ≥1 своё поле.** Тип только со встроенными полями бессмыслен.
5. **Встроенные поля задачи** (всегда есть, не описываются в типе): `title` (≤250,
   обязателен), `deadline` (обязателен), `type`. `mxboard_field` описывает только
   ДОПОЛНИТЕЛЬНЫЙ контент.
6. **Значения полей — JSON** на задаче (это формализованный контент). Метаданные
   (стадия, автор, исполнитель, дедлайн) — настоящие колонки, по ним фильтруем/сортируем.
7. **Стадии — проектные.** Ключ `done` не захардкожен: правило закрытия завязано на
   флаг `is_final`. Инвариант: у проекта ровно одна начальная и ровно одна финальная.
   `stage_key` (необязательный сквозной тег) даёт отчётность через проекты — так стадия
   становится «общей».
8. **Подзадачи вместо ревьюеров** (роль ревьюера сочли слишком узкой; наблюдателей
   тоже нет). Привлечь кого-то к работе = завести **подзадачу**: это обычная задача
   с `parent_id`, живёт в `mxboard_task` (отдельной таблицы нет). Создать подзадачу
   может **и автор, и исполнитель** основной. Незавершённая (не в финальной стадии)
   подзадача — **блокер**: основную нельзя перевести в финальную, пока открыт хоть
   один потомок.
9. **Видимость**: вижу задачу, если я автор / исполнитель, либо автор / исполнитель
   любой её подзадачи (это даёт чтение + комментирование основной), либо
   супер-пользователь группы её проекта, либо глобальный sudo. Основная при этом
   **не попадает в канбан** соисполнителя подзадачи — у него в доске только сама
   подзадача (его задача).
10. **Дедлайн оспаривается, не меняется исполнителем.** Исполнитель предлагает новую
    дату с причиной; автор принимает (дедлайн меняется) или отклоняет.
11. **MCP: `tools/list` фильтруется по правам токена.** Исполнитель видит ~6 тулов,
    менеджер — весь набор. Тулы дискретные (лучшая читаемость/валидация), но каждый
    агент несёт только свой срез — контекст не раздувается.
12. **Файлы**: значение поля-файла в JSON (`path`, `name`, `size`); заливка в UI через
    медиа-источник MODX (`mxboard.media_source`); агент через MCP пока задаёт URL.

## Права исполнителя (жёстко)

| Действие | Исполнитель | Автор / супер-группы / sudo |
|---|---|---|
| Двигать до финальной (не в неё) | да | да |
| Перевести в финальную (закрыть) | нет | да (если нет открытых подзадач) |
| Создать подзадачу | да | да |
| Комментировать | да | да |
| Оспорить дедлайн | да | — |
| Принять/отклонить оспаривание | нет | да |
| Править поля / тип / дедлайн / приоритет | нет | да |
| Удалить задачу | нет | да |

«Менеджер» = супер-пользователь группы отдела или глобальный sudo (тот же признак,
что в видимости).

## Модель данных (9 таблиц)

- **mxboard_department** — `usergroup_id` (группа MODX), `name` (переопределение), `active`, `position`, `createdon`. Реестр: помечает группу как отдел.
- **mxboard_project** — `department_id`, `key`, `name`, `description`, `active`, `position`, `createdon`, `updatedon`. Уникум `(department_id, key)`.
- **mxboard_column** — `project_id`, `key`, `name`, `position`, `move_roles`, `stage_key` (nullable), `is_initial`, `is_ready`, `is_final`, `createdon`. `project_id=0` — глобальный шаблон. Уникум `(project_id, key)`.
- **mxboard_task_type** — `department_id`, `key`, `name`, `description`, `active`, `position`, `createdon`. «Рабочий» только при ≥1 поле. Уникум `(department_id, key)`.
- **mxboard_field** — `task_type_id`, `key`, `label`, `type` (text/textarea/url/number/date/select/user/file), `required`, `position`, `options` (JSON), `createdon`. Уникум `(task_type_id, key)`.
- **mxboard_task** — метаданные: `project_id`, `parent_id` (nullable — если задан, задача является подзадачей; тот же проект, что у родителя), `type_id`, `column_id`, `author_id`, `assignee_id`, `priority`, `position`, `deadlineon`, `deadline_disputed`, `deadline_proposed`, `createdon`, `updatedon`, `startedon`, `closedon`; контент: `title` (≤250), `tor` (опц.), `fields` (JSON), `meta` (JSON). Индекс по `parent_id`.
- **mxboard_comment** — `task_id`, `user_id`, `content`, `createdon`. Без изменений.
- **mxboard_log** — `task_id`, `user_id`, `action`, `from_column`, `to_column`, `note`, `channel`, `createdon`. Новые action: `deadline_dispute`, `deadline_accepted`, `deadline_rejected`, `subtask_add`.
- **mxboard_token** — `user_id`, `name`, `token_hash` (sha256), `active`, `lastusedon`, `createdon`. Без изменений.

## Инварианты (проверяются на сервере)

- Проект: **ровно одна** `is_initial`, **ровно одна** `is_final`; ≥1 `is_ready` — предупреждение.
- Тип: **≥1** поле, иначе не «рабочий» (не в `task_schema`/`task_create`).
- Задача: `type_id` обязателен и валиден; `title` непустой ≤250; `deadlineon` > 0.
- В финальную стадию (по флагу `is_final`) — только автор/супер/sudo.
- В финальную **нельзя, пока есть незавершённая подзадача** (потомок не в своей финальной стадии) — блокер.
- Самозакрытие (автор=исполнитель) запрещено, если `mxboard.allow_self_close=0`.
- Подзадача (`parent_id` задан) наследует проект родителя; `parent_id` должен ссылаться на существующую задачу того же проекта.

## Механика оспаривания дедлайна

- Исполнитель: `task_dispute_deadline(task_id, proposed_date, reason)` → `deadline_disputed=1`,
  `deadline_proposed=date`, причина в журнал (`deadline_dispute`), событие.
- Автор: `task_resolve_dispute(task_id, accept)` →
  - accept: `deadlineon=deadline_proposed`, флаг и proposed сброшены, журнал `deadline_accepted`;
  - reject: флаг и proposed сброшены, дедлайн прежний, журнал `deadline_rejected`.

## MCP — дискретные тулы, `tools/list` по правам токена

**Чтение** (кто видит доску): `board_list`, `task_get` (отдаёт родителя и список подзадач), `task_schema`, `project_list`.
**Жизненный цикл** (член отдела): `task_create` (создавший=автор; опц. `parent_id` → подзадача, разрешено только автору/исполнителю родителя), `task_take`, `task_move` (блок финальной при открытых подзадачах), `task_comment`, `task_dispute_deadline`.
**Авторские** (автор/супер/sudo): `task_update`, `task_resolve_dispute`, `task_delete`.
**Структура** (супер/sudo — агент-менеджер): `type_create`/`type_update`, `project_create`/`project_update`, `department_register`.

- `type_create` принимает поля в том же вызове (инвариант ≥1 поле).
- `project_create` принимает колонки (инвариант одна initial / одна final) либо берёт шаблон.
- `task_schema` отдаёт полный обязательный набор: встроенные (title, deadline) + обязательные поля типа.

## Системные настройки

| Ключ | Умолчание | Смысл |
|---|---|---|
| `mxboard.default_project` | `default` | проект по умолчанию |
| `mxboard.api_enabled` | `1` | REST включён |
| `mxboard.mcp_enabled` | `1` | MCP включён |
| `mxboard.allow_self_close` | `0` | разрешить закрывать свою же задачу |
| `mxboard.wip_limit` | `0` | лимит задач в работе на исполнителя |
| `mxboard.group_admin_authority` | `0` | супер группы, если `role.authority <= порог` (меньше = больше прав; `0` = выключено) |
| `mxboard.media_source` | `0` | медиа-источник для файлов (0 = дефолтный) |

---

## Что НЕ трогаем (работает)

Атомарный захват (`UPDATE ... WHERE assignee_id=0`), журнал, сброс OPcache в деплое,
поле `position` (не `rank`), гвард `hasPermission` без сессии (`Transitions::isSuperuser`).

- [x] Механика «супер-пользователь группы» в MODX 3 (`modUserGroupMember.role` → `modUserGroupRole.authority`) — сверено по codegraph (эталон MODX 3). **Итог:** `authority` (int unsigned, default 9999), **меньше = больше прав**; `role=0` = роли нет.
