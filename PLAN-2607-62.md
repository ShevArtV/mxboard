# План: очереди задач (карточка #77 / 2607-62)

Статус: подготовлен на стадии `to_start`, ожидает приёмки Менеджером. Код не начинать до «go».

Ограничение оператора: **без релизного цикла**. Только план → реализация → локальные проверки →
runtime-выгрузка на стенд Hostland (`modx3.art-sites.ru`). Никаких bump версии, changelog как
релизного артефакта, сборки transport, modstore, публичной доки, прод-обновлений.

## 1. Что установлено ресёрчем (факты, на которых стоит план)

**Стадии — не хардкодить, брать из модели.** У колонки три флага (`schema/mxboard.mysql.schema.xml:85-91`):
`is_initial` (начальная, куда падает новая карточка), `is_start` (стартовая, с неё идёт отсчёт
факта), `is_final` (финальная = закрытие). На проекте `mxboard` это `backlog` / `to_start` / `done`
— ровно те три роли, которыми оперирует ToR. Значит:
- «добавить в очередь можно только из начальной стадии» → `is_initial`;
- «перетащить в стартовую стадию» и «следующая уезжает в стартовую» → `is_start`.

**Единая точка перемещения** — `TaskService::move()` (`src/Service/TaskService.php:208-272`):
права (`Transitions::can`), блокер подзадач, журнал (`log()`), события
`mxbOnBeforeTaskMove` / `mxbOnTaskMove` / `mxbOnTaskClose` (`fireEvent()`, :1179-1199), учёт времени.
Автоперемещение обязано идти через него — иначе обойдём журнал, права и события.

**События** — `fireEvent()` шлёт `invokeEvent()` + `NotificationService::emit()`. Payload
формируется в одном месте (:1182-1186) → ключ `queue` добавляется там же, один раз, для всех событий.
Внешняя лента `/events` (`BoardQuery::events():376-448`) и живой SSE-поток
(`liveEvents():459-528`) строятся из `mxboard_log` с уже готовым JOIN на `Task` — ключ `queue`
туда добавляется одним полем в `select()`.

**Права структуры** — `Transitions::isDepartmentManager($modx, $user, $departmentId)`
(`src/Helpers/Transitions.php:185`); так гейтятся все структурные операции в `StructureService`
(напр. `updateProject():577`). «Кто может управлять проектом» = менеджер отдела проекта.

**Процессоры** — тонкие обёртки над сервисом (`Processors/Mgr/ServiceProcessor.php`), фронт зовёт
их по FQCN через `api/connector.js`.

**Фронт** — Vite 6, вход `main-board.js` → `vue-dist/board.min.js|css`; `vue`/`primevue` external
(import map VueTools). Важно: канбан живёт в **`pages/BoardView.vue` (461 стр.)**, а `BoardApp.vue`
— оболочка табов + глобальный CSS. Панель фильтров — `BoardView.vue:356-414`. DnD самописный,
нативный HTML5 (`onDragStart:277`, `onDrop:304-337`), оптимистичный с откатом. Диалога
подтверждения на drop сейчас нет ни одного — писать с нуля по образцу `confirm.require()` из
`deleteTask()` (`BoardView.vue:233-253`). Живое обновление — SSE `board-event` → `bus.js` →
дебаунс-`load()` (`BoardView.vue:212-225`).

**Тестов нет** (только `tests/ExampleTest.php`). Проверки в проекте делаются in-process
smoke-скриптами на стенде: `_proc_smoke_remote.php`, `_api_smoke_remote.php`, `_time_smoke_remote.php`.

## 2. Проектные решения (и почему так)

### 2.1. Членство в очереди — поля в `mxboard_task`, без таблицы связки

`mxboard_queue` (новая таблица) + два поля в `mxboard_task`: `queue_id` (0 = не в очереди) и
`queue_position`.

Почему не таблица `mxboard_queue_item`: ToR говорит «задача **находится в** очереди», событие несёт
**один** `queue` id — то есть задача максимум в одной очереди. При полях в задаче этот инвариант
бесплатен, а ключ `queue` в `/events`, `liveEvents` и `fireEvent` берётся из уже загруженной задачи
без единого лишнего запроса и без нового JOIN. Таблица связки понадобилась бы только под «много
очередей на задачу» или историю членства — ни того, ни другого ToR не требует.

Цена решения: при удалении очереди надо руками обнулить `queue_id` у её задач (composite-связь
поля не чистит) — делается одним `updateCollection` в сервисе.

### 2.2. Завершённая задача остаётся в очереди

`queue_id` при закрытии **не** обнуляется — иначе события после закрытия потеряют ключ `queue`
(а AC п.18 требует его во всех связанных событиях). «Следующая в очереди» ищется среди задач с
`closedon = 0`; в аккордеоне показываются только незакрытые.

### 2.3. Автозапуск — вызов из `TaskService::move()`, не плагин на событие

После успешного входа в финальную колонку `move()` зовёт `QueueService::advance()`, который делает
обычный `TaskService::move()` для следующей задачи. Значит автоперемещение проходит полный штатный
путь: права, журнал, `mxbOnTaskMove`, уведомления (AC п.16, 17).

Почему не плагин пакета на `mxbOnTaskClose`: плагин можно отключить/не установить, порядок
обработчиков не гарантирован, а в CLI-смоуках его пришлось бы поднимать отдельно. Требование ToR
«через штатные системные события» — про то, что автодействие обязано **порождать** события, а не
про способ подписки; через `move()` оно их порождает.

Защита от рекурсии: статический guard-флаг в `QueueService` (авто-move идёт не в финальную колонку,
но флаг ставим явно). Канал журнала для автодействия — `queue`, чтобы в логе автозапуск отличался
от ручного перевода.

### 2.4. Кандидат на автозапуск

Первая по `queue_position` задача очереди, у которой `closedon = 0` **и** текущая колонка
`is_initial`. Если такой нет — очередь просто ничего не запускает (тихо, с записью в лог MODX).
Так автозапуск не может утащить назад задачу, которую уже двигали руками.

### 2.5. Права

| Действие | Кто |
|---|---|
| CRUD очередей (`Структура → Очереди`) | менеджер отдела проекта (`isDepartmentManager`) — AC п.4 |
| Добавить/убрать задачу в очередь | автор задачи или менеджер отдела |
| Изменить порядок drag-and-drop | менеджер отдела проекта |
| Старт очереди перетаскиванием в стартовую | обычные права перехода (`Transitions::can`); переупорядочивание — следствие разрешённого move, отдельного права не требует |
| Просмотр аккордеона очередей | кто видит доску |

### 2.6. Что добавляю сверх буквы ToR (и зачем)

- **`key` у очереди** (уникален в рамках проекта, автогенерация из названия) — все прочие сущности
  пакета адресуются ключом; без него агент по MCP не сможет сослаться на очередь иначе как по id.
- **Действие «убрать из очереди»** — иначе ошибочно поставленную задачу нельзя вынуть, а очередь
  начнёт запускать не то. Реализуется тем же диалогом.

## 3. Этапы работ

### Этап 1. Модель и миграции

1. `core/components/mxboard/schema/mxboard.mysql.schema.xml` — объект `MxBoardQueue`
   (таблица `mxboard_queue`): `project_id`, `key`, `name`, `description`, `active`, `position`,
   `createdon`, `updatedon`; unique-индекс `(project_id, key)`, индексы `project_id`, `position`;
   `aggregate Project`. **Зачем:** очередь — сущность проекта (AC п.3).
2. Там же в `MxBoardTask` — поля `queue_id` (int, index, default 0) и `queue_position`
   (int, default 0). **Зачем:** членство и порядок (2.1).
3. `core/components/mxboard/src/Model/MxBoardQueue.php` + `src/Model/mysql/MxBoardQueue.php` —
   сгенерированные классы модели (по образцу `MxBoardProject`); правка
   `src/Model/mysql/MxBoardTask.php` под новые поля. **Зачем:** xPDO без классов таблицу не увидит.
4. `package_builder/packages/mxboard/resolvers/01.resolve.tables.php` — `MxBoardQueue::class` в
   `$classes` и `ALTER`-миграции `mxboard_task.queue_id|queue_position` в `$migrations`.
   **Зачем:** установка/обновление пакета на любом сайте (даже если transport собираем не сейчас —
   код обязан быть консистентным).
5. Новый `_queue_migrate_remote.php` в корне репо (по образцу `_time_migrate_remote.php`).
   **Зачем:** выгрузка на стенд идёт **без** сборки transport — таблицу и поля на стенде создаёт
   этот одноразовый идемпотентный скрипт.

### Этап 2. Сервис очередей

6. Новый `core/components/mxboard/src/Service/QueueService.php`:
   - `list(modUser, int $projectId, bool $withTasks)` — очереди проекта (+ незакрытые задачи с
     `num`/`title`/`queue_position`); **зачем:** данные для вкладки структуры и аккордеона (AC п.2, 10).
   - `create/update/remove(modUser, ...)` — гейт `isDepartmentManager`, уникальность `key` в проекте,
     при удалении — обнуление `queue_id`/`queue_position` у задач очереди (AC п.4, 2.1).
   - `addTask(modUser, int $taskId, int $queueId)` — гейт «автор или менеджер», проверка
     `column.is_initial` у задачи с внятной ошибкой (AC п.7), проверка «очередь того же проекта»,
     `queue_position` = хвост очереди.
   - `removeTask(modUser, int $taskId)` — вынуть задачу из очереди (2.6).
   - `reorder(modUser, int $queueId, array $orderedTaskIds)` — гейт менеджера, полная перестановка
     по образцу `StructureService::reorderColumns():989` (AC п.11).
   - `promote(int $taskId)` — сделать задачу первой в очереди, остальные сдвинуть (AC п.14);
     вызывается при старте не-первой задачи.
   - `advance(MxBoardTask $closed, string $channel)` — найти кандидата (2.4) и перевести его в
     колонку `is_start` через `TaskService::move()` от имени **автора этой задачи** (AC п.15, 16).
   - `queueIdOf(int $taskId)` / `isFirst(int $taskId)` — вспомогательные для payload событий и для
     проверки «первая ли» перед стартом.
7. Лексиконы ошибок сервиса — `core/components/mxboard/lexicon/{ru,en}/default.inc.php`
   (`mxboard_err_queue_not_found`, `mxboard_err_queue_not_initial`, `mxboard_err_queue_denied`,
   `mxboard_err_queue_key_exists`, `mxboard_err_queue_no_start_column`, …). **Зачем:** сервис
   возвращает лексиконные ключи, оба файла обязательны (иначе «сырой» ключ в UI).

## 4. Риски

| Риск | Митигация |
|---|---|
| Автозапуск зациклится или задвоит перевод | guard-флаг в `QueueService`, кандидат только из `is_initial`, `advance` в `try/catch` |
| У проекта не задана стартовая стадия (`is_start`) | внятная ошибка `mxboard_err_queue_no_start_column`, автозапуск молча пропускается с записью в лог MODX |
| Автор следующей задачи не вправе двигать её в стартовую | `move()` вернёт отказ; логируем предупреждение, очередь не двигается — молчаливого обхода прав не делаем |
| Самописный DnD хрупкий, порядка внутри колонки нет | DnD очереди делаем **внутри аккордеона**, отдельным списком, не трогая DnD колонок |
| Забытый ключ в `lexicon/en` | правка обоих файлов в одном коммите, проверка в браузере |
| Миграция на стенде без transport | идемпотентный `_queue_migrate_remote.php`, повторный запуск безопасен |

## 5. Оценка

План карточки — 12 ч. По разбивке этапов выходит ~16 ч (backend ~6, UI ~5, проверки ~3, выгрузка и
отчёт ~2). Оценка оспорена штатно через `task_dispute_plan`; решение за автором карточки.
