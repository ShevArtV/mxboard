2.4.1-pl
==============
- NEW: Стадия «Старт» (to_start) между «Бэклогом» и «Планом» — ручной гейт запуска
  исполнителя. Постановка вызревает в бэклоге, и только автор (move_roles=author)
  открывает работу переводом карточки в «Старт». Это важно, когда исполнитель — ИИ-агент:
  внешняя автоматизация будит его строго с этой стадии, а не в момент создания задачи,
  поэтому можно готовить и согласовывать постановку заранее, а старт назначать отдельно.
  Второй гейт («Согласование») остаётся: реализация начинается только после одобрения плана.
  Добавлена также тупиковая стадия «Отменена» (cancel) — не финальная, закрытием считается
  только «Готово». Резолвер теперь подтягивает position и move_roles у уже существующих
  колонок, иначе при вставке стадии в середину набора порядок на доске разъезжается.
- NEW: Тип поля «Список» (select) заработал с обоих концов: в форме поля появился
  редактор вариантов (через `|`), а в карточке задачи такое поле рендерится выпадающим
  списком. Раньше select можно было выбрать в интерфейсе, но вариантов задать было негде,
  и поле показывалось обычным текстовым вводом.
- NEW: Тип задачи «Ресёрч» (research) в поставке — изучить вопрос и вернуть ответ.
- FIX: Поле типа без явного `type` больше не отваливается с «Недопустимый тип поля».
  Значение по умолчанию — `text` — отсутствовало в списке разрешённых типов, то есть
  дефолт заведомо не проходил собственную валидацию (StructureService::FIELD_TYPES).
- FIX: `options` поля не сохранялись из интерфейса — ключ отсутствовал в белом списке
  свойств процессоров Field/Create и Field/Update (та же ловушка, что с `color` у колонки).
- CHANGE: REST GET /events отдаёт `task_stage` — ключ текущей стадии карточки. У события
  `move` стадия видна по from/to, но у `create` и `comment` её не было вовсе, и внешняя
  автоматизация не могла решить, стоит ли будить исполнителя.
- CHANGE: MCP type_create описывает фактический список типов поля
  (text|textarea|url|number|date|select|user|files) — раньше рекламировался
  несуществующий `file` и умалчивались `text` и `select`.

2.4.0-pl
==============
- BUILD: Первый публичный релиз на modstore. Документация — docs.modx.pro/components/mxboard,
  исходники — github.com/ShevArtV/mxboard, лицензия MIT.
- NEW: Иконка удаления (корзина) в правом верхнем углу карточки на доске — можно удалить
  задачу, не открывая её. Появляется на наведении (на тач-экранах видна всегда), спрашивает
  подтверждение у самой кнопки. Видна только тому, кто вправе удалить (автор или менеджер);
  права всё равно проверяет сервер (TaskService::delete).
- CHANGE: Доска по умолчанию заточена под работу с агентами — шесть стадий вместо четырёх:
  backlog → plan → confirm → in_progress → review → done. Исполнитель выносит план работ в
  «План» (move_roles=assignee), и только автор одобряет его переводом в «Подтверждение»
  (move_roles=author); не согласен — комментирует, стадию не меняет. Дальше как раньше:
  in_progress/review — исполнитель, done (final) — только автор. Backlog больше не принимает
  возврат рядовыми ролями (move_roles пуст — откат только менеджером). Изменения в резолвере
  02.resolve.board.php (глобальный шаблон колонок и проект default). Логика переходов на ключи
  колонок не завязана (работает через is_initial/is_final), существующие установки резолвер не
  трогает — новая раскладка применяется на чистой установке.
- FIX: move_roles больше не поддерживает роль any: рядовой участник доски не должен
  двигать карточку только по факту доступа. Менеджерский обход остаётся отдельной
  проверкой (sudo или супер-пользователь группы отдела).
- NEW: Кнопка «Сбросить к дефолтным» в редакторе стадий (Структура) — удаляет собственные
  колонки проекта и возвращает его на глобальный шаблон. Сам шаблон (project_id=0) сбросить
  нельзя. Задачи не осиротеют: сервис в транзакции переносит карточки на одноимённую (по key)
  колонку шаблона, а несовпавшие ключи — на стартовую (is_initial) стадию.
- CHANGE: Редактор стадий у проекта без своих колонок больше не показывает read-only список
  дефолтов — вместо него плашка «Своих колонок нет — используются дефолтные» с подсказкой
  скопировать шаблон. На канбане дефолты остаются как fallback (задачи физически стоят в
  стадиях шаблона — иначе их негде показать).
- FIX: Колокольчик уведомлений больше не «выпадает» из белой полосы табов на серый фон.
  Фон и нижняя граница перенесены с TabList на весь ряд .mxb-tabbar — полоса читается
  единой белой поверхностью, включая зону колокольчика справа.

2.3.0-alpha
==============
- NEW: In-app уведомления в админке через SSE (Server-Sent Events). Колокольчик со
  счётчиком непрочитанных в шапке доски + всплывающие тосты. Событие жизненного цикла
  задачи (создание, смена стадии, комментарий, оспаривание/решение дедлайна) пишется в
  очередь mxboard_notification на участников задачи (автор+исполнитель) за вычетом того,
  кто действие совершил. Поток читает эндпоинт assets/components/mxboard/sse.php через
  нативный EventSource (авторизация по mgr-куке; соединение короткое, докачка по
  Last-Event-ID). Клик по уведомлению открывает задачу.
  Настройки: mxboard.sse_enabled (вкл/выкл фичу целиком), mxboard.sse_lifetime (сек),
  mxboard.sse_poll_interval (сек). Новая таблица mxboard_notification.
- NEW: REST-эндпоинт GET /events?since=<log_id>&limit= — инкрементальная лента журнала
  с курсором по id, обогащённая полями задачи (num/title/project/автор/исполнитель) и
  именем актора. Только менеджеру отдела. Точка интеграции для внешних потребителей
  (напр. локальный поллер, дёргающий агентов через Jarvis).

2.2.0-alpha
==============
- NEW: Киоск-доступ «только канбан». При установке создаётся политика доступа
  «mxBoard Only» (whitelist поверх AdministratorTemplate: frames, load, list, view,
  home, components, logout) — назначьте её группе-отделу через «Доступ к контексту mgr»,
  и её рядовые члены увидят в менеджере только доску (без деревьев ресурсов/элементов/
  файлов и прочих разделов). Права урезаются политикой, поэтому «видно кнопку» не значит
  «есть доступ». Политика создаётся один раз и никому не назначается автоматически.
- NEW: Плагин mxBoardKiosk (OnManagerPageInit) + настройка mxboard.kiosk_usergroups
  (CSV имён групп): рядового члена такой группы при входе редиректит сразу на канбан.
  По умолчанию пусто — никого не трогает (opt-in). Sudo не запирается.

2.1.0-alpha
==============
- UX: Поле «кто может двигать карточку в колонку» — вместо ручного CSV теперь радио:
  Автор и исполнитель / Только автор / Только исполнитель (в БД по-прежнему move_roles).
- FIX: Канбан не помещал длинные колонки — нижние карточки уходили под низ фрейма
  менеджера без прокрутки. Колонка теперь ограничена высотой вьюпорта, карточки
  скроллятся внутри колонки (шапка стадии остаётся на месте).
- FIX: Реактивная синхронизация вкладок «Структура» и канбана без перезагрузки страницы:
  новый проект сразу доступен во вкладке «Колонки»; изменение состава/порядка/цвета колонок
  отражается на доске автоматически (лёгкая шина ревизий utils/bus.js).
- CHANGE: Проекты больше не копируют колонки автоматически при создании. Проект без своих
  колонок показывает дефолтный шаблон (project_id=0) только для чтения (fallback на доске,
  в stage_list и при создании задач).
- NEW: Ручное копирование колонок в проект (кнопка «Копировать колонки») из другого проекта
  того же отдела или дефолтного шаблона; если источник один — без диалога. Доступно только
  пока в проекте нет ни одной задачи (процессор Column/Copy, Column/Sources).
- NEW: Переупорядочивание колонок drag-n-drop в редакторе стадий (процессор Column/Reorder).
- NEW: Человекочитаемый номер задачи num (напр. 2607-15) — уникальный адрес наравне с id;
  MCP/REST (task_get/move/update/... и /tasks/{num}) принимают id ИЛИ num. Формат в настройке
  mxboard.task_num_format (по умолчанию {y}{m}-{num}); счётчик атомарный по периоду
  (таблица mxboard_counter), устойчив к удалению задач. Существующим задачам num проставляется
  бэкофиллом при обновлении.
- CHANGE: Убрано неиспользуемое поле колонки stage_key (тег стадии) из схемы, API и UI.
- FIX: При клонировании колонок из шаблона терялся цвет (color) — теперь переносится.


1.0.0-alpha
==============
- NEW: Канбан-доска для ИИ-агентов. Доски, настраиваемые колонки, карточки с ToR (markdown).
- NEW: Модель прав относительно карточки: у задачи есть автор и исполнитель. Право перехода
  задаётся на колонке (move_roles: author / assignee / group:Name).
  В финальную колонку (done) переводит только автор задачи.
- NEW: Атомарный захват задачи (UPDATE ... WHERE assignee_id = 0) — гонку двух агентов
  выигрывает ровно один, второй получает внятный отказ.
- NEW: Настройка mxboard.allow_self_close (по умолчанию 0): запрет закрывать задачу,
  где автор и исполнитель — один пользователь (защита от самоаттестации агента).
- NEW: MCP-эндпоинт (JSON-RPC 2.0, Streamable HTTP) — агенты подключаются по URL + Bearer-токен,
  локально ничего ставить не нужно. Пять инструментов: board_list, task_create, task_take,
  task_move, task_comment.
- NEW: Токены агентов: в базе хранится только sha256-хэш, сам токен показывается один раз.
- NEW: Журнал переходов (кто, когда, откуда, куда, каким каналом) — пишется процессорами,
  из интерфейса не редактируется.
- NEW: События mxbOnTaskCreate / Take / Release / BeforeTaskMove / Move / Close / Comment
  и произвольные метаданные карточки (meta) — точки интеграции с внешними системами.
- NEW: Интерфейс доски в менеджере (Vue 3 + PrimeVue через VueTools), drag&drop.
