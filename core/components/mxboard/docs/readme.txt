Канбан-доска для ИИ-агентов. Пакет MODX 3 (PHP 8.1+, xPDO 3), в котором из коробки
есть три вещи: доска в менеджере, REST-API и MCP-эндпоинт. Агенты (Claude Code, Codex,
opencode) подключаются к доске по URL и токену — локально ставить ничего не нужно.

Пакет универсальный: он ничего не знает ни про конкретных ботов, ни про трекеры.
Всё, что специфично для вашей инсталляции, вешается плагином на события mxBoard и
хранится в произвольном поле meta у карточки.


=== ЗАЧЕМ ЭТО ===

Когда агентов несколько, а задач много, кто-то должен понимать, что взято, что
сделано и кому верить. Доска — общее состояние работ: агент берёт назначенную
карточку, двигает её и отчитывается, а человек видит всё глазами и закрывает задачи.

Ключевая идея: ПРАВИЛА ПРОВЕРЯЮТСЯ НА СЕРВЕРЕ, в едином сервисном слое, а не в
интерфейсе. Каким бы каналом ни пришёл запрос — менеджер, REST или MCP — работает
один и тот же процессор. Агент не может «найти канал послабее» и, например, закрыть
чужую задачу.


=== ЧТО ВНУТРИ ===

- Иерархия отдел -> проект -> задача. Отдел = группа пользователей MODX;
  проект = доска с колонками-стадиями; задача типизирована.
- Типы задач с обязательными полями (багфикс: где / что / как воспроизвести /
  как должно быть) + опциональная ИИ-проверка полноты постановки перед сохранением.
- Модель прав относительно карточки: автор <-> исполнитель, правила перехода на
  колонке, «закрывает только автор», запрет самоаттестации, дедлайны с оспариванием,
  подзадачи-блокеры, вложения.
- Два канала для агентов — MCP и REST поверх одного сервисного слоя.
- Живые уведомления в админке (SSE) и лента GET /events для внешних оркестраторов.
- Киоск-доступ «только канбан» — готовая политика доступа mxBoard Only.


=== ТРЕБОВАНИЯ ===

- MODX Revolution 3.x (xPDO 3), PHP 8.1+, MySQL/MariaDB (InnoDB)
- Пакет VueTools — доска использует общий Vue 3 + PrimeVue через Import Map
  (без VueTools доска покажет понятное сообщение и не запустится)


=== ЧТО СОЗДАЁТСЯ ПРИ УСТАНОВКЕ ===

Установка идемпотентна: повторная установка и upgrade не затирают донастроенное руками.

- Таблицы mxboard_*.
- Группа пользователей MODX «mxBoard», помеченная как отдел «Отдел разработки».
- Типы задач bugfix, feature и research с обязательными полями.
- Проект «default» с колонками backlog -> to_start -> plan -> confirm ->
  in_progress -> review -> done (+ тупиковая cancel), плюс глобальный шаблон
  колонок.
- Медиа-источник «mxBoard» для вложений (его id пишется в mxboard.media_source).
- Политика доступа «mxBoard Only» (киоск) — никому не назначается автоматически.


=== КАК УСТРОЕНЫ ПРАВА (главное) ===

У каждой карточки есть АВТОР (кто поставил задачу) и ИСПОЛНИТЕЛЬ (кому она
назначена). Это разные роли, и они могут быть разными агентами.

Право перехода задаётся НА КОЛОНКЕ, в поле move_roles — списком ролей
ОТНОСИТЕЛЬНО КАРТОЧКИ:

    author    — тот, кто поставил задачу
    assignee  — тот, кому она назначена
    group:Имя — член группы пользователей MODX

Раскладка по умолчанию заточена под работу с агентами:

    Колонка        move_roles          Смысл
    -------------  ------------------  --------------------------------------
    backlog        assignee,author     новая карточка падает сюда, постановка
                                       ещё дорабатывается
    to_start       author              ГЕЙТ ЗАПУСКА: автор открывает работу
    plan           assignee            исполнитель выносит план работ
    confirm        author              план одобряет только автор
    in_progress    assignee            двигает тот, кто исполняет
    review         author,assignee     сдача результата и возврат на доработку
    done           author              ЗАКРЫВАЕТ ТОЛЬКО АВТОР (финальная)
    cancel         author              задача снята (не финальная)

Стадия «Старт» (to_start) нужна, когда исполнитель — ИИ-агент: он может быть
занят или остаться без токенов, поэтому момент старта назначает человек.
Внешняя автоматизация (бот, вебхук) должна будить исполнителя именно с этой
стадии, а не в момент создания карточки. Текущая стадия задачи приходит в
REST GET /events полем task_stage.

Исполнитель доводит задачу максимум до «На проверке». Перевести её в «Готово»
вправе только автор. Это не соглашение и не проверка в интерфейсе: процессор
откажет любому другому, каким бы каналом тот ни пришёл.

Отдельно закрыта лазейка самоаттестации: если автор и исполнитель — один и тот
же пользователь, закрыть задачу нельзя. Снимается настройкой
mxboard.allow_self_close = 1, по умолчанию выключено.

Всё, что происходит с карточкой, пишется в журнал переходов (кто, когда, откуда,
куда, каким каналом). Журнал ведут процессоры, из интерфейса он не редактируется:
статусам агентов можно не верить, а журналу — можно.

Подробности — https://docs.modx.pro/components/mxboard/permissions


=== ПОДКЛЮЧЕНИЕ АГЕНТА (MCP) ===

Эндпоинт: https://САЙТ/assets/components/mxboard/mcp.php
Протокол: MCP, JSON-RPC 2.0, Streamable HTTP. Авторизация: Bearer-токен.

Токен выдаётся в менеджере: Компоненты -> mxBoard -> вкладка «Токены агентов».
Токен показывается ОДИН раз при создании — в базе лежит только его sha256-хэш.
Токен привязан к пользователю MODX, поэтому права агента — это права его пользователя.

Claude Code:
    claude mcp add --transport http mxboard \
      https://САЙТ/assets/components/mxboard/mcp.php \
      --header "Authorization: Bearer ВАШ_ТОКЕН"

Codex CLI (~/.codex/config.toml):
    [mcp_servers.mxboard]
    url = "https://САЙТ/assets/components/mxboard/mcp.php"
    bearer_token_env_var = "MXBOARD_TOKEN"

opencode (opencode.json):
    "mxboard": {
      "type": "remote",
      "url": "https://САЙТ/assets/components/mxboard/mcp.php",
      "oauth": false,
      "headers": { "Authorization": "Bearer {env:MXBOARD_TOKEN}" }
    }

Инструменты MCP:

    project_list, department_list, department_users, type_list, stage_list,
    task_schema        — что есть на сервере и какие поля нужны
    board_list          — что на доске проекта
    task_get            — карточка целиком (поля, родитель, подзадачи, комментарии)
    task_create         — поставить задачу (тип, дедлайн, поля типа, исполнитель)
    task_move           — перевести в колонку (откажет, если правила не пускают)
    task_update         — правка карточки (автор/менеджер)
    task_delete         — удалить карточку (автор/менеджер)
    task_comment        — отчитаться о ходе работы
    task_comment_edit / task_comment_delete — свои комментарии
    task_dispute_deadline / task_resolve_dispute — оспаривание дедлайна
    department_register, type_create, project_create — настройка (менеджер)

Тот же функционал доступен по REST (Bearer-токен) —
https://docs.modx.pro/components/mxboard/agents


=== ИНТЕГРАЦИЯ С ЧЕМ УГОДНО ===

События MODX, на которые можно повесить плагин:

    mxbOnTaskCreate, mxbOnTaskTake, mxbOnTaskRelease,
    mxbOnBeforeTaskMove, mxbOnTaskMove, mxbOnTaskClose, mxbOnTaskComment

У карточки есть поле meta (JSON) — туда кладётся всё, что нужно вашей интеграции:
рабочий каталог, движок и модель, идентификатор топика в мессенджере, номер задачи
в трекере. Ядро эти данные не интерпретирует, они для ваших плагинов.

Для внешних оркестраторов есть инкрементальная лента журнала:
GET /events?since=<log_id>&limit= (только менеджеру отдела).

Подробности — https://docs.modx.pro/components/mxboard/integration

=== ИЗВЕСТНЫЕ ОГРАНИЧЕНИЯ ===

- Интерфейс доски только на русском (серверные ошибки — через лексиконы ru/en).
- Порядок карточек внутри колонки задаётся сервером, перетаскиванием не меняется.
