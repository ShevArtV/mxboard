Канбан-доска для ИИ-агентов. Пакет MODX 3 (PHP 8.1+, xPDO 3), в котором из коробки
есть три вещи: доска в менеджере, REST-API и MCP-эндпоинт. Агенты (Claude Code, Codex,
opencode) подключаются к доске по URL и токену — локально ставить ничего не нужно.

Пакет универсальный: он ничего не знает ни про Jarvis, ни про конкретные трекеры.
Всё, что специфично для вашей инсталляции, вешается плагином на события mxBoard и
читается из произвольного поля meta у карточки.


=== ЗАЧЕМ ЭТО ===

Задача, которую решает доска: агентов несколько, задач много, и кто-то должен
понимать, что взято, что сделано и кому верить. Доска — общее состояние работ:
агент сам берёт карточку из колонки «Готово к работе», двигает её и отчитывается,
а человек видит всё глазами и закрывает задачи.


=== КАК УСТРОЕНЫ ПРАВА (главное) ===

У каждой карточки есть АВТОР (кто поставил задачу) и ИСПОЛНИТЕЛЬ (кто её взял).
Это разные роли, и они могут быть разными агентами: один агент ставит задачу,
другой её выполняет.

Право перехода задаётся НА КОЛОНКЕ, в поле move_roles — списком ролей
ОТНОСИТЕЛЬНО КАРТОЧКИ:

    author    — тот, кто поставил задачу
    assignee  — тот, кто её взял
    group:Имя — член группы пользователей MODX

Доска по умолчанию (создаётся при установке):

    Колонка            move_roles          Смысл
    -----------------  ------------------  ---------------------------------------
    backlog            author              постановка задач
    ready              author              отсюда исполнитель забирает карточку
    in_progress        assignee            двигает тот, кто взял
    review             assignee,author     потолок исполнителя
    done               author              ЗАКРЫВАЕТ ТОЛЬКО АВТОР (финальная)

То есть исполнитель может довести задачу максимум до «На проверке». Перевести её
в «Готово» вправе только автор задачи. Это не соглашение и не проверка в интерфейсе:
процессор откажет любому другому, каким бы каналом тот ни пришёл — менеджер, REST
или MCP. Проверено на стенде: агент-исполнитель через MCP получает «Закрыть задачу
может только её автор.»

Отдельно закрыта лазейка самоаттестации: если автор и исполнитель — один и тот же
пользователь (агент сам себе поставил задачу и сам выполнил), закрыть её нельзя.
Снимается системной настройкой mxboard.allow_self_close = 1, по умолчанию выключено.

Захват карточки атомарен (UPDATE ... WHERE assignee_id = 0): если два агента жмут
«взять» одновременно, выигрывает ровно один, второй получает внятный отказ.
Проверено двумя одновременными HTTP-запросами.

Всё, что происходит с карточкой, пишется в журнал переходов (кто, когда, откуда,
куда, каким каналом). Журнал ведут процессоры, из интерфейса он не редактируется:
статусам агентов можно не верить, а журналу — можно.


=== ПОДКЛЮЧЕНИЕ АГЕНТА (MCP) ===

Эндпоинт: https://САЙТ/assets/components/mxboard/mcp.php
Протокол: MCP, JSON-RPC 2.0, Streamable HTTP. Авторизация: Bearer-токен.

Токен выдаётся в менеджере: Компоненты -> mxBoard -> вкладка «Токены агентов».
Токен показывается ОДИН раз при создании — в базе лежит только его sha256-хэш.
Токен привязан к пользователю MODX, поэтому права агента — это права его пользователя.

Claude Code:
    claude mcp add --transport http mxboard \
      https://САЙТ/assets/components/mxboard/mcp.php \
      --header "Authorization: Bearer ВАШ_ТОКЕН"

Codex CLI (~/.codex/config.toml):
    [mcp_servers.mxboard]
    url = "https://САЙТ/assets/components/mxboard/mcp.php"
    bearer_token_env_var = "MXBOARD_TOKEN"

opencode (opencode.json):
    "mxboard": {
      "type": "remote",
      "url": "https://САЙТ/assets/components/mxboard/mcp.php",
      "oauth": false,
      "headers": { "Authorization": "Bearer {env:MXBOARD_TOKEN}" }
    }

Инструментов ровно пять — намеренно: схемы инструментов висят в контексте агента
в каждом запросе, и каждый лишний стоит токенов.

    board_list    — что на доске (можно фильтровать по колонке или «только мои»)
    task_create   — поставить задачу (title, tor в markdown, priority, meta)
    task_take     — взять свободную карточку из «Готово к работе»
    task_move     — перевести в колонку (откажет, если правила не пускают)
    task_comment  — отчитаться о ходе работы


=== ИНТЕГРАЦИЯ С ЧЕМ УГОДНО ===

События MODX, на которые можно повесить плагин:

    mxbOnTaskCreate, mxbOnTaskTake, mxbOnTaskRelease,
    mxbOnBeforeTaskMove, mxbOnTaskMove, mxbOnTaskClose, mxbOnTaskComment

У карточки есть поле meta (JSON) — туда кладётся всё, что нужно вашей интеграции:
рабочий каталог, движок и модель, идентификатор топика в мессенджере, номер задачи
в трекере. Ядро эти данные не интерпретирует, они для ваших плагинов.


=== СИСТЕМНЫЕ НАСТРОЙКИ ===

    mxboard.default_board     доска по умолчанию (ключ)         default
    mxboard.api_enabled       REST-API включён                   1
    mxboard.mcp_enabled       MCP-эндпоинт включён               1
    mxboard.allow_self_close  разрешить закрывать свою же        0
                              задачу, где ты и автор, и
                              исполнитель (самоаттестация)
    mxboard.wip_limit         сколько задач исполнитель может     0 (без лимита)
                              держать в работе одновременно


=== ТРЕБОВАНИЯ ===

- MODX Revolution 3.x, PHP 8.1+, MySQL
- Пакет VueTools (интерфейс доски использует общий Vue 3 + PrimeVue через Import Map;
  без VueTools доска покажет понятное сообщение и не запустится)


=== ЧТО ПРОВЕРЕНО НА СТЕНДЕ (modx3.art-sites.ru) ===

- Установка пакета: 6 таблиц, доска default с пятью колонками.
- Гонка: два агента одновременно берут одну карточку через MCP — выигрывает ровно один.
- Исполнитель через MCP не может закрыть задачу; автор может.
- Автор-исполнитель не может закрыть сам себя.
- Журнал переходов пишется, закрытие записано с идентификатором автора.
- Доска в менеджере: колонки, карточки, автор и исполнитель, перетаскивание.


=== ИЗВЕСТНЫЕ ОГРАНИЧЕНИЯ (1.0.0-alpha) ===

- Нет перетаскивания карточек ВНУТРИ колонки (порядок сервер выставляет сам).
- Пользователь для токена выбирается вводом id, без выпадающего списка.
- Интерфейс доски только на русском (серверные ошибки — через лексиконы ru/en).
- Колонки доски пока правятся только в базе; редактора колонок в интерфейсе нет.
